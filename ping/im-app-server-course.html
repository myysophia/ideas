应用全貌

这个 IM 后端到底做了什么？让我们从用户的视角开始。

想象一个超级邮局

你每天用的IM 应用，背后有一个庞大而精密的服务器系统在运转。

想象一个邮局——不是收发普通信件的那种，而是一个能让上千人同时实时交流的超级邮局。它不仅要瞬间送达消息，还要管理群组、处理图片、发送推送通知。

💡
核心能力

这个后端服务的核心任务只有一件事：让消息安全、快速、可靠地在用户之间流动。所有看似复杂的功能——群组、@提及、已读回执——都是围绕这个核心展开的。

四大功能支柱

💬

单聊私聊

两个人之间的一对一实时对话，消息瞬间送达，支持文本、图片、文件。

👥

群组聊天

最多数百人的群组对话，支持权限管理、成员角色、@提及通知。

📎

媒体分享

图片、视频、文件的上传、压缩、存储。文件存在 AWS S3 上，只传链接。

🔔

推送通知

用户不在线时，通过 Firebase 发送手机推送通知，确保不错过重要消息。

02

五层架构

像一栋五层楼，每层只做自己该做的事，绝不越界。

分层架构：请求的生命周期

想象一栋写字楼。1 楼是前台（接收请求），2 楼是安检（验证身份），3 楼是档案室（查数据），4 楼是仓库（存数据），5 楼是外部联络处（调用第三方服务）。每层只能和相邻楼层沟通。

💡
核心规则

Controller 只能调用 Repository，Repository 只能调用 Model。反向依赖是被严格禁止的——就像你不能让档案室去指挥前台。

目录结构 = 楼层地图

看看代码是怎么按层组织的：

src/ 整个后端代码都在这里
api/ 接入层 — 接收请求、返回响应
middleware/ 中间件层 — 安检：鉴权、限流、验证
model/access/ 数据访问层 — 档案室：封装所有数据库查询
user.repo.js用户数据查询
conversation.repo.js对话数据查询
group.repo.js群组数据查询
model/postgres/ 持久化层 — PostgreSQL 数据表定义
model/dynamodb/ 持久化层 — DynamoDB 表定义
model/redis/ 持久化层 — Redis 缓存数据
lib/services/ 服务层 — 外部服务集成（Ably、S3、SQS）

03

身份验证

像高级会所的门卫：先看邀请函，再核对指纹。

两道安检

用户登录后，服务器发给他一个 JWT（电子身份证）。之后每次请求，都要过两道安检：

1
检查 JWT 签名

验证这个"身份证"是服务器签发的，没有被篡改，也没有过期。就像检查身份证上的防伪标志。

2
查 Redis 确认有效

即使 JWT 没过期，用户可能已经登出了。Redis 里存着"仍然有效的 token"列表。就像确认访客还在会所内。

3
放行并标记身份

两道都通过后，在请求上下文中标记用户 ID、设备 ID、租户 ID。后续处理直接读取这些信息。

认证代码解读

CODE 
const header = ctx.header.authorization;
const token = header.replace("Bearer ", "");
const decoded = jwt.verify(token, secretKey);
const isValid = await redis.get(
  `device:token:${decoded.userId}:${decoded.deviceId}`
);
if (!isValid) throw new BaseError(
  BizErrorCode.TokenInvalid
);
ctx.currentUserId = decoded.userId;
ctx.deviceId = decoded.deviceId;
语文

从请求头中取出 Authorization 字段

去掉 "Bearer " 前缀，得到纯 token

用密钥验证 JWT 签名，解出里面的用户信息

用用户ID和设备ID组合成 key，去 Redis 查这个 token

Redis 里的 key 格式：device:token:用户ID:设备ID

如果 Redis 里没有，说明 token 已失效

抛出 TokenInvalid 错误，请求被拒绝

验证通过，把用户 ID 存到请求上下文中

同时保存设备 ID——用于设备级别的控制

💡
设备级 JWT

每个 JWT 都绑定了特定设备。你在手机上登录不会影响电脑的登录——因为它们的 token 不同（deviceId 不同）。这就是为什么你可以在手机和电脑上同时在线，互不干扰。

群消息的两阶段处理

群消息的发送和接收看似一气呵成，实际上分成了"快路径"和"慢路径"两条流水线：

1
用户发送群消息

客户端将消息直接发布到 Ably 对应的群组频道。

2
消息实时到达所有群成员

通过 Ably 频道，所有在线成员几乎同时收到消息。

3
DynamoDB 快速持久化消息

快路径：消息写入 DynamoDB，确保聊天记录立即可查。

4
SQS 队列接收处理任务

慢路径：消息处理任务被放入 SQS 队列，等待后台消费。

5
后台消费者处理重任务

@提及通知、未读计数更新、推送通知——这些耗时操作全部异步完成，不影响消息到达速度。

多数据库策略

一个项目，三个数据库 — 各司其职，各取所长

为什么一个项目需要三个数据库？

📚

像一个图书馆系统：PostgreSQL 是卡片目录（复杂关联），DynamoDB 是书架（海量但结构简单），Redis 是你的书签（临时、快速、丢了也没关系）。

🏛️
PostgreSQL — 关系型数据库
存储用户、群组、对话等有关系的数据。像 Excel 表格，用外键把不同表连起来。
⚡
DynamoDB — 文档数据库
存储消息。每秒能处理上万次写入，不需要复杂查询。
🧠
Redis — 内存数据库
存储临时数据：设备 token、会话缓存、限流计数。重启就清空，所以只存不重要的。

代码翻译 — 创建对话（PostgreSQL 事务）

CODE
async function createDirectConversation(userId, targetId, tenantId) {
  const t = await sequelize.transaction();

  try {
    const conversation = await Conversation.create({
      type: "direct",
      tenantId
    }, { transaction: t });

    await Participant.bulkCreate([
      {
        conversationId: conversation.id,
        userId,
        tenantId
      },
      {
        conversationId: conversation.id,
        userId: targetId,
        tenantId
      }
    ], { transaction: t });

    await UserConversation.bulkCreate([
      {
        userId,
        conversationId: conversation.id,
        tenantId
      },
      {
        userId: targetId,
        conversationId: conversation.id,
        tenantId
      }
    ], { transaction: t });

    await t.commit();
    return conversation;

  } catch (err) {
    await t.rollback();
    throw err;
  }
}
译文

开启一个数据库事务——要么全部成功，要么全部失败。

创建对话记录（类型：一对一）。

批量创建两个参与者（自己和对方）。

为每个用户创建用户-对话关联记录。

全部成功，提交事务。

任何一步失败，回滚所有操作。

精妙设计 — 群组与对话的关系

💡
精妙设计

群组没有独立的成员表。群组的所有成员管理和聊天功能都委托给底层的 Conversation 来处理。这减少了代码重复，也意味着"创建群组"本质上就是"创建一个带成员的对话"。

错误处理 — 当事情出错时

优雅地处理异常，把混乱变成有序反馈

错误处理：优雅地说"出错了"

想象一家餐厅。如果你点了不存在的菜，服务员不会把厨房烧了——她会说"这道菜暂时没有，可以试试别的"。这就是错误处理的本质：把混乱变成有序的反馈。

BaseError 类

class BaseError extends Error {
  constructor(biz, err_msg = "") {
    super(err_msg);
    this.code = typeof biz === "number" ? biz : biz.code;
    this.err_msg = err_msg || biz.msg;
  }
}

BaseError 继承自 JavaScript 内置的 Error 类

构造函数接收一个业务错误码和一个可选的消息

自动提取错误码数字

如果没有提供消息，使用默认的

错误码范围

100000-199999 用户相关错误：登录失败、账号不存在、Token 过期
200000-299999 群组相关错误：群不存在、无权限、已满员
300000-399999 会议相关错误：会议不存在、已结束
800000-899999 媒体相关错误：文件过大、格式不支持

统一错误处理

// ctx.state.error — 统一错误响应
ctx.state.error = (err) => {
  if (err instanceof BaseError) {
    ctx.status = 200;
    ctx.body = { code: err.code, err_msg: err.err_msg };
  } else {
    logger.error("Unexpected error", err);
    ctx.status = 200;
    ctx.body = { code: HttpErrorCode.ServerError, err_msg: "服务异常" };
  }
};

定义统一的错误处理方法

如果是已知的业务错误，返回具体的错误码和消息

如果是未知错误，记录日志但只返回通用的"服务异常"

HTTP 状态码始终返回 200——业务错误码在 body 里区分

⚠️

注意：HTTP 状态码始终是 200，即使出错。这是 REST API 的一种设计选择——让客户端统一解析 body 中的 code 字段来判断成功与否，而不是同时检查 HTTP 状态码和 body。这样更简单，但也有人认为不够"RESTful"。

找出 Bug

下面代码中有一行有 bug。你能找到吗？

app.get("/profile", async (ctx) => {
  const user = getUserProfile(ctx.currentUserId);
  ctx.body = ctx.state.ok(user);
});
💡

提示

getUserProfile 是一个 async 函数，返回的是 Promise 而不是实际数据。

答案：第2行缺少 await。
应该是：const user = await getUserProfile(ctx.currentUserId);

否则返回的是一个 Promise 对象，而不是用户数据！

关键架构决策

🚀

Client-Direct-Publish

消息不经过服务器直接到达。优势：极致实时。代价：服务器无法拦截/审核消息。

⚡

Two-Phase Processing

快速路径存消息，慢速路径处理业务。优势：消息即时到达。代价：如果 SQS 失败，后续处理可能延迟。

🗄️

Three Databases

按数据特性选数据库。优势：每个数据库做最擅长的事。代价：运维复杂度增加。

🔐

Bitmask Permissions

用二进制位表示权限。优势：极其紧凑，一个数字存所有权限。代价：可读性差，调试不直观。

权限位掩码系统

const PERMISSIONS = {
  SCREENSHOT_ALLOWED: 0,    // 第0位：允许截图
  ALL_SPEAK: 1,             // 第1位：全员禁言开关
  PROFILE_VIEW_ALLOWED: 2,  // 第2位：查看资料
};

class GroupPermissionChecker {
  hasPermission(permissionStr, permissionIndex) {
    return permissionStr[permissionIndex] === "1";
  }
}

定义每种权限对应的二进制位位置

用字符串索引直接检查某位是否为1

例如："110" 表示第0位=1(截图)、第1位=1(全员可发言)、第2位=0(不能看资料)

💎

软件工程没有"完美方案"，只有"最适合当前场景的方案"。每个设计决策都是权衡：你要实时性，就要接受服务器无法审核消息。你要简单，就要接受一个数字代表所有权限。理解这些权衡，就是理解架构设计的核心。
